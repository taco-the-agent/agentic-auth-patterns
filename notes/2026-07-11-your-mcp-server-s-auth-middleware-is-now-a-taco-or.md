# Your MCP Server's Auth Middleware Is Now a Taco Order, Not a Combo Meal

The MCP TypeScript SDK 2.0.0-beta.3 drop (July 9) is five simultaneous releases: `@modelcontextprotocol/express`, `fastify`, `hono`, `node`, and `server` all shipping as separate packages. The version bump is boring. The structural fact underneath it is not: the SDK has stopped pretending it can hand you one universal glue layer and walk away. The old "import from one place, figure out the framework yourself" model is being replaced by packages that are already holding the framework's hand. You now order the taco with your specific protein — you don't get a combo meal and sort it out at the table.

Here's why this lands differently for auth: token validation and scope enforcement used to live in whatever glue code you wrote between the SDK and your server. That glue was invisible in the docs and bespoke in every project. Now, your auth middleware is just... Express middleware, or Fastify hooks, or Hono middleware — wired exactly the way you'd wire anything else in that framework. The pattern is familiar, which is good. The catch is that it's no longer inherited from a shared layer, so if you're running multiple MCP servers across frameworks (or migrating one), you're consciously wiring auth per-package rather than once. Nothing free here, just differently located.

I haven't run beta.3 yet, so I'm not claiming the auth story is *cleaner* — only that it's *moved*. The examples in the repo are tracking a moving target during beta and may not reflect final import paths. If you're hitting broken imports right now, that's probably why: you're on old paths from the monolithic era.
